Plan de Pruebas QA — Event Publisher Kafka

Plan de pruebas basado en los escenarios BDD definidos para el starter de publicación automática de eventos en Kafka.
Contexto
El objetivo de este plan es validar el JAR del starter Event Publisher Kafka como un producto, verificando su comportamiento real en un proyecto Spring Boot de prueba, siguiendo los escenarios BDD definidos.
Precondiciones generales
  • Kafka accesible (local o en contenedor) en localhost:9092 o equivalente.
  • Proyecto Spring Boot de prueba con el starter incluido como dependencia.
  • Topics creados: qa-topic y qa-dead-letter (o los configurados en propiedades).
  • Servicio de prueba con métodos anotados con @PublishEvent.
  • Herramientas de observación: kafka-console-consumer y acceso a logs de la aplicación.

Casos de prueba basados en BDD

ID Escenario BDD Pasos de prueba (QA) Resultado esperado
TC-EP-001 BDD-EP-001
Método anotado finaliza correctamente.
  • Arrancar Kafka y la aplicación de prueba.
  • Invocar el método okEvent() anotado con @PublishEvent(topic="qa-topic").
  • Escuchar el topic qa-topic con kafka-console-consumer.
  • Se recibe un mensaje JSON en el topic qa-topic.
  • Los logs muestran:
    • Programación de publicación asíncrona.
    • Intento de publicación.
    • Confirmación de publicación correcta.
  • El método retorna sin bloqueos apreciables.
TC-EP-002 BDD-EP-002
Método anotado devuelve null.
  • Invocar el método nullEvent() anotado con @PublishEvent(topic="qa-topic").
  • Escuchar el topic qa-topic durante unos segundos.
  • No se recibe ningún mensaje en qa-topic.
  • Los logs muestran que el método devolvió null y no se publicará evento.
  • No se registran reintentos.
TC-EP-003 BDD-EP-003
Método anotado lanza excepción.
  • Invocar el método exceptionEvent() anotado con @PublishEvent.
  • Observar el comportamiento de la aplicación y los logs.
  • Escuchar el topic qa-topic.
  • El método lanza una excepción hacia el llamador.
  • No se publica ningún mensaje en qa-topic.
  • Los logs indican que el método lanzó excepción y no se publicará evento.
TC-EP-004 BDD-EP-004
Error temporal al enviar evento (reintentos).
  • Configurar event.publisher.retry.max-attempts=3.
  • Simular fallo temporal de Kafka (por ejemplo, parar Kafka tras arrancar la app).
  • Invocar okEvent().
  • Arrancar Kafka antes de que terminen todos los reintentos (opcional).
  • Los logs muestran varios intentos de publicación (hasta 3).
  • Si Kafka se recupera antes del último intento, el mensaje acaba publicándose.
  • El método de negocio no se bloquea durante los reintentos (publicación asíncrona).
TC-EP-005 BDD-EP-005
Error permanente → envío a dead-letter.
  • Configurar event.publisher.dead-letter-topic=qa-dead-letter.
  • Parar Kafka o usar un topic inaccesible.
  • Invocar okEvent().
  • Escuchar el topic qa-dead-letter cuando Kafka esté disponible.
  • Tras agotar los reintentos, no se publica en qa-topic.
  • Se publica un mensaje en qa-dead-letter con:
    • payload original
    • topic original
    • número de reintentos
    • timestamp
    • mensaje y tipo de error
TC-EP-006 BDD-EP-006
Reprocesar mensajes desde dead-letter.
  • Generar mensajes en qa-dead-letter (por ejemplo, con TC-EP-005).
  • Arrancar Kafka y la aplicación con el listener de dead-letter activo.
  • Observar logs y consumir mensajes de qa-topic.
  • Los mensajes de qa-dead-letter se consumen y se reenvían a qa-topic.
  • Los logs muestran el reproceso y el resultado de cada mensaje.
TC-EP-007 BDD-EP-007
Configuración externa desde application.properties.
  • Modificar propiedades de event.publisher.* (reintentos, dead-letter topic, logger).
  • Reiniciar la aplicación de prueba.
  • Repetir algunos casos de prueba (por ejemplo, TC-EP-001 y TC-EP-005).
  • El comportamiento del componente cambia según la configuración sin recompilar el JAR.
  • Los nuevos valores se reflejan en los logs y en los topics usados.
TC-EP-008 BDD-EP-008
Logging dedicado en texto plano.
  • Configurar event.publisher.logger.name y el fichero de log en el sistema de logging.
  • Ejecutar varios métodos anotados (okEvent, nullEvent, exceptionEvent).
  • Revisar consola y fichero de log.
  • Se generan entradas de log en consola y en el fichero dedicado.
  • Los mensajes están en texto plano, con información de:
    • eventos publicados
    • reintentos
    • errores
    • envíos a dead-letter
Este plan de pruebas puede ampliarse con automatización (JUnit + Testcontainers + Kafka) reutilizando los mismos casos.
