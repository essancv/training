COMPONENTE ARQUITECTURA · EVENT PUBLISHER KAFKA

BDD — Escenarios Gherkin

Escenarios de comportamiento expresados en Gherkin para validar el componente de publicación automática de eventos Kafka.
ID Escenario (Gherkin)
BDD-EP-001 
Feature: Publicación automática de eventos Kafka

  Scenario: Método anotado finaliza correctamente
    Given un método anotado con @PublishEvent(topic="alta-solicitada")
    And el método devuelve un objeto válido
    When el método finaliza sin lanzar excepción
    Then el componente debe serializar el objeto a JSON
    And debe publicarlo asíncronamente en el topic "alta-solicitada"
    And el hilo del método no debe bloquearse
BDD-EP-002 
  Scenario: Método anotado devuelve null
    Given un método anotado con @PublishEvent(topic="alta-solicitada")
    And el método devuelve null
    When el método finaliza correctamente
    Then no debe publicarse ningún evento
    And no deben generarse reintentos
BDD-EP-003 
  Scenario: Método anotado lanza excepción
    Given un método anotado con @PublishEvent(topic="alta-solicitada")
    And el método lanza una excepción
    When el componente intercepta la ejecución
    Then no debe publicarse ningún evento
    And el error debe registrarse en el logger dedicado
BDD-EP-004 
  Scenario: Error temporal al enviar evento
    Given un evento válido que debe enviarse a Kafka
    And el envío falla temporalmente
    And el número de reintentos configurado es 3
    When el componente intenta enviar el evento
    Then debe reintentarlo 3 veces de forma asíncrona
    And si uno de los reintentos tiene éxito
    Then el evento debe considerarse enviado correctamente
BDD-EP-005 
  Scenario: Error permanente tras agotar reintentos
    Given un evento válido que debe enviarse a Kafka
    And el envío falla en todos los reintentos
    And el dead-letter topic está configurado
    When se agotan los reintentos
    Then el evento debe enviarse al dead-letter topic
    And el mensaje debe incluir payload original, topic original, número de reintentos, timestamp y causa del fallo
BDD-EP-006 
  Scenario: Reintento desde dead-letter topic
    Given un mensaje almacenado en el dead-letter topic
    And el proceso de reintentos está activo
    When el componente consume el mensaje
    Then debe intentar reenviarlo al topic original
    And debe registrar el resultado en el logger dedicado
BDD-EP-007 
  Scenario: Configuración externa desde application.properties
    Given un proyecto que usa el componente
    And existen propiedades definidas en application.properties
    When el componente arranca
    Then debe cargar bootstrap servers, reintentos, dead-letter topic y configuración de logging
    And no debe requerir configuración adicional en código
BDD-EP-008 
  Scenario: Logging dedicado
    Given el componente está configurado con un fichero de log específico
    And el nivel de log está definido en application.properties
    When se publica un evento, se reintenta o se produce un error
    Then el componente debe escribir un log en texto plano en el fichero configurado
    And también debe escribir en consola
Estos escenarios BDD pueden usarse directamente para automatizar pruebas de aceptación del componente.
