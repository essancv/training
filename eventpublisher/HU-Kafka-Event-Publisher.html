COMPONENTE ARQUITECTURA · EVENT PUBLISHER KAFKA

Historias de Usuario — Event Publisher Kafka

Backlog funcional del componente reutilizable de publicación automática de eventos en Kafka, pensado para equipos de desarrollo y plataforma.
ID Rol Historia Descripción / Alcance Impacto Prioridad
HU-EP-001 Desarrollador Como desarrollador
quiero publicar eventos Kafka usando una anotación
para no tener que escribir código Kafka en cada proyecto. 		El componente debe exponer una anotación @PublishEvent(topic="...") que, aplicada sobre un método, publique automáticamente el valor de retorno en el topic indicado cuando el método finalice sin excepción. Si el método devuelve null o lanza una excepción, no se publicará ningún evento. Impacto: elimina boilerplate de integración con Kafka y reduce errores de implementación en proyectos de negocio. ALTA
HU-EP-002 Desarrollador Como desarrollador
quiero que la publicación de eventos sea asíncrona
para que el flujo de negocio no se bloquee por problemas de Kafka. 		El componente debe enviar los eventos a Kafka de forma asíncrona, sin bloquear el hilo del método anotado. El método debe devolver su resultado inmediatamente, independientemente del éxito o fallo del envío. La gestión de reintentos y errores será responsabilidad interna del componente. Impacto: protege el rendimiento y la experiencia de usuario de los servicios de negocio ante latencias o caídas de Kafka. ALTA
HU-EP-003 Arquitecto Como arquitecto
quiero que el componente se integre como un starter auto-configurable
para que los equipos solo tengan que añadir una dependencia. 		El componente debe empaquetarse como un Spring Boot Starter, con auto-configuración basada en la presencia de Kafka y propiedades en application.properties. No debe requerir configuración manual adicional en cada proyecto, más allá de las propiedades de entorno. Impacto: facilita la adopción masiva del componente y reduce la variabilidad entre proyectos. ALTA
HU-EP-004 Equipo Plataforma Como equipo de plataforma
quiero que todos los eventos se serialicen en JSON de forma estándar
para garantizar consistencia entre servicios. 		El componente debe serializar todos los eventos a JSON usando un ObjectMapper interno, con configuración estándar. La estructura del JSON debe ser estable y homogénea para todos los proyectos que usen el componente. Impacto: simplifica la interoperabilidad entre servicios y el análisis de eventos en herramientas externas. MEDIA
HU-EP-005 Equipo Plataforma Como equipo de plataforma
quiero que el componente implemente reintentos automáticos de envío
para reducir la pérdida de eventos ante fallos temporales. 		El componente debe reintentar el envío de eventos fallidos de forma asíncrona, un número de veces configurable desde application.properties. Los reintentos no deben bloquear el flujo de negocio y deben estar desacoplados del método anotado. Impacto: mejora la resiliencia frente a fallos puntuales de infraestructura sin complejidad adicional en los servicios de negocio. ALTA
HU-EP-006 Equipo Plataforma Como equipo de plataforma
quiero que los eventos no entregados se envíen a un dead-letter topic
para poder analizarlos y reprocesarlos posteriormente. 		Si tras agotar los reintentos un evento no puede enviarse al topic original, el componente debe publicarlo en un dead-letter topic configurable. El mensaje debe incluir payload original, topic original, número de reintentos, timestamp y causa del fallo. Impacto: permite trazabilidad y recuperación de eventos problemáticos sin perder información crítica. ALTA
HU-EP-007 Equipo Plataforma Como equipo de plataforma
quiero que el componente incluya un proceso de reintentos desde el dead-letter topic
para poder reintentar la entrega de eventos fallidos. 		El componente debe incluir un consumidor propio que lea del dead-letter topic y reintente enviar los mensajes al topic original. Este proceso será asíncrono, configurable y formará parte del mismo componente de arquitectura. Impacto: habilita estrategias de recuperación sin necesidad de desarrollar procesos ad hoc en cada proyecto. MEDIA
HU-EP-008 Arquitecto Como arquitecto
quiero que toda la configuración del componente se gestione desde application.properties
para centralizar la gestión por entorno. 		El componente debe obtener toda su configuración (bootstrap servers, número de reintentos, backoff, dead-letter topic, logger, fichero de log, nivel de log) desde application.properties o equivalentes por entorno, sin requerir código adicional. Impacto: facilita la operación en múltiples entornos (dev, pre, pro) y reduce el riesgo de configuraciones divergentes. MEDIA
HU-EP-009 Equipo Plataforma Como equipo de plataforma
quiero que el componente tenga un logger dedicado en texto plano
para poder auditar fácilmente la actividad de publicación de eventos. 		El componente debe usar un logger propio, con nombre configurable, que escriba en consola y en un fichero específico. El formato será texto plano y el nivel de log será configurable desde propiedades. Debe registrar eventos publicados, reintentos, errores y envíos a dead-letter topic. Impacto: mejora la observabilidad y facilita el diagnóstico de problemas relacionados con la mensajería. MEDIA
HU-EP-010 Arquitecto Como arquitecto
quiero que el componente funcione en modo best effort sin transacciones
para simplificar la primera versión y reducir complejidad inicial. 		La primera versión del componente no estará integrada con transacciones de base de datos ni implementará el patrón outbox. La publicación de eventos será best effort: el flujo de negocio no se verá bloqueado ni revertido por fallos en Kafka. Impacto: permite entregar valor rápido con un componente usable, dejando la integración transaccional para una evolución posterior. BAJA
Estas historias de usuario pueden servir como base para derivar SBE, ATDD y BDD del componente de publicación de eventos Kafka.
