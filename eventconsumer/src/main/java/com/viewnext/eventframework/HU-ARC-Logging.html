Historias de Usuario y Arquitectura - Infraestructura de Eventos

1. Historias de Usuario del Componente de Eventos

1. Logging de publicación de eventosLogging

Historia de usuario Criterios de aceptación
Como desarrollador o equipo de observabilidad, quiero que el componente registre trazas detalladas al publicar eventos para que pueda monitorizar, depurar y auditar el comportamiento del sistema.

Dado que se publica un evento

Cuando comienza la publicación

Entonces debe registrarse el tipo del evento y su payload (o payload enmascarado si está activado el flag).

Dado que la publicación finaliza correctamente

Cuando el evento se envía al broker

Entonces debe registrarse una traza de éxito.

Dado que ocurre un error durante la publicación

Cuando se lanza una excepción

Entonces debe registrarse una traza de error con detalles del fallo.

2. Logging de consumo de eventosLogging

Historia de usuario Criterios de aceptación
Como desarrollador o equipo de observabilidad, quiero que el componente registre trazas detalladas al consumir eventos para que pueda monitorizar, depurar y auditar el comportamiento del sistema.

Dado que se recibe un evento desde Kafka

Cuando comienza su procesamiento

Entonces debe registrarse el tipo del evento y su payload (o payload enmascarado).

Dado que el evento se procesa correctamente

Cuando el método consumidor finaliza

Entonces debe registrarse una traza de éxito.

Dado que ocurre un error durante el consumo

Cuando se lanza una excepción

Entonces debe registrarse una traza de error y activarse la lógica de reintentos o DLQ.

3. Configuración del loggingConfiguración

Historia de usuario Criterios de aceptación
Como aplicación que usa el componente, quiero poder configurar el nivel de detalle del logging y si el payload debe enmascararse para que pueda cumplir requisitos de seguridad y auditoría.

Dado un archivo de configuración de la aplicación

Cuando se define eventframework.logging.mask-payload=true

Entonces el componente debe ocultar el contenido del payload en logs.

Dado que no se define ninguna configuración

Entonces el componente debe usar valores por defecto seguros.

2. Decisiones de Arquitectura

Framework: Spring Boot
Motivo: integración natural con Kafka, AOP, DI y configuración.
Broker inicial: Kafka
Motivo: estándar de la industria, soporte nativo en Spring, alto rendimiento.
Nota: la arquitectura sigue desacoplada para permitir otros brokers en el futuro.
Serialización: JSON con Jackson
Motivo: estándar en Spring, flexible, soporta polimorfismo.
Deserialización: Basada en campo type dentro del mensaje
Ejemplo JSON: 
{
  "type": "SystemStatusEvent",
  "version": 1,
  "status": "OK",
  "message": "Todo correcto"
}
Motivo: permite múltiples tipos de eventos en un mismo topic.
Concurrencia: Thread pool propio
Motivo: desacoplar la ejecución del consumidor del listener de Kafka y permitir paralelismo configurable.
Gestión de errores: DLQ + reintentos configurables
Motivo: resiliencia, trazabilidad y control de fallos.
Versionado de eventos: Campo version en cada evento
Motivo: permite evolución del modelo sin romper consumidores antiguos.
Logging del componente: Logging propio y configurable
Motivo: separar claramente las trazas de la infraestructura de eventos de las trazas de la aplicación.
Incluye:
  • Inicio y fin de publicación de eventos
  • Inicio y fin de consumo de eventos
  • Errores en publicación y consumo
  • Flag para enmascarar payload

3. Arquitectura General

🟦 Capa de Dominio

  • Entidades de dominio
  • Eventos de dominio (POJOs con campo version)
  • Interfaces de contratos de publicación y consumo

🟩 Capa de Aplicación

  • Casos de uso
  • Servicios de orquestación
  • Puertos de entrada y salida

🟧 Capa de Infraestructura

  • Anotaciones @PublishEvent y @ConsumeEvent
  • Aspectos AOP
  • Escáner de anotaciones
  • Registro de consumidores
  • Adaptadores Kafka (producer/consumer)
  • Thread pool propio para ejecución paralela
  • Serialización/deserialización JSON con Jackson
  • DLQ + reintentos
  • Logging propio del componente (publicación y consumo)

🟫 Capa de Adaptadores

  • Controladores REST
  • Beans que publican eventos
  • Beans que consumen eventos
