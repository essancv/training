Framework de Eventos - Documentación Final

1. Visión general

Este framework proporciona una infraestructura de eventos basada en Kafka, diseñada con Clean Architecture y SOLID, que permite:

  • Publicar eventos de dominio sin acoplarse a Kafka ni a la infraestructura.
  • Consumir eventos mediante anotaciones declarativas.
  • Registrar trazas detalladas de publicación y consumo.
  • Configurar enmascaramiento de payloads sensibles.
  • Gestionar reintentos y envío a DLQ.
  • Serializar y deserializar eventos con soporte polimórfico mediante el campo type.

El objetivo es que las aplicaciones de negocio trabajen con POJOs de dominio y casos de uso, mientras la infraestructura se encarga de Kafka, logging, concurrencia y resiliencia.

3. Decisiones de arquitectura

Framework base: Spring Boot
Integración natural con Kafka, AOP, DI y configuración externa.
Broker inicial: Kafka
Estándar de la industria, soporte nativo en Spring, alto rendimiento. La arquitectura se mantiene desacoplada para permitir otros brokers.
Serialización: JSON con Jackson
Se añade el campo type para soportar deserialización polimórfica. 
{
  "type": "SystemStatusEvent",
  "version": 1,
  "status": "OK",
  "message": "Todo correcto"
}
Concurrencia: Thread pool propio en EventListenerContainer
Permite paralelismo configurable y desacopla el listener de Kafka del procesamiento real.
Gestión de errores: Reintentos + DLQ
Reintentos configurables y envío a un topic de DLQ mediante KafkaDlqPublisher.
Logging: Componente centralizado EventLogger
Registra inicio/fin de publicación y consumo, errores y permite enmascarar payloads.
Registro de tipos de eventos: EventTypeRegistry
Permite resolver el tipo de evento a partir del campo type del JSON.

4. Arquitectura por capas

4.1 Dominio

DomainEvent SystemStatusEvent EventConsumerHandler

Contiene los eventos de dominio y contratos puros, sin dependencias de infraestructura.

  • DomainEvent: interfaz base de eventos.
  • SystemStatusEvent: ejemplo de evento de estado.
  • EventConsumerHandler<T>: contrato para manejadores de eventos.

4.2 Aplicación

EventPublisher PublishEventUseCase ProcessConsumedEventUseCase

Orquesta la lógica de publicación y procesamiento de eventos.

  • EventPublisher: puerto de salida para publicar eventos.
  • PublishEventUseCase: caso de uso para publicar eventos.
  • ProcessConsumedEventUseCase: caso de uso para procesar eventos consumidos.

4.3 Infraestructura

@PublishEvent @ConsumeEvent PublishEventAspect ConsumeEventScanner ConsumerRegistry KafkaEventPublisher KafkaEventListener EventListenerContainer EventSerializer EventDeserializer EventLogger KafkaDlqPublisher

Implementa la integración con Kafka, logging, concurrencia y serialización.

4.4 Configuración

EventFrameworkConfig LoggingConfigProperties EventTypeRegistry

Ensamblan todos los componentes en el contexto Spring.

5. Flujo de publicación y consumo

5.1 Publicación de eventos

  1. Método de aplicación anotado con @PublishEvent(topic = "...").
  2. PublishEventAspect intercepta el método.
  3. Se obtiene el DomainEvent devuelto.
  4. EventLogger registra EVENT-PUBLISH-START.
  5. PublishEventUseCase delega en EventPublisher.
  6. KafkaEventPublisher serializa con EventSerializer e incluye type.
  7. Se envía el mensaje a Kafka.
  8. EventLogger registra EVENT-PUBLISH-SUCCESS o EVENT-PUBLISH-ERROR.

5.2 Consumo de eventos

  1. Kafka entrega un ConsumerRecord al contenedor.
  2. EventListenerContainer lo envía a un thread del pool.
  3. KafkaEventListener obtiene los consumidores del ConsumerRegistry.
  4. Se parsea el JSON y se lee el campo type.
  5. EventDeserializer o ObjectMapper crean el DomainEvent correcto.
  6. EventLogger registra EVENT-CONSUME-START.
  7. ProcessConsumedEventUseCase aplica lógica transversal y llama al método consumidor.
  8. EventLogger registra EVENT-CONSUME-SUCCESS o EVENT-CONSUME-ERROR.
  9. Si hay errores, EventListenerContainer aplica reintentos y, si se agotan, envía a DLQ con KafkaDlqPublisher.
